完整 Findings 清单

维度 1：系统状态可见性

BLOCKER 浇水页（watering）JS 与 WXML 数据绑定不匹配，页面永远显示空状态 BUG ▸
问题描述
watering.wxml 使用 {{pendingDecisions}} 数组和 onGoReview 事件渲染待复盘决策列表，但 watering.js 的 data 中只有 decision: null（单个对象），方法名是 onSkipWatering / onSubmitWatering。结果是 WXML 的 wx:for 遍历 undefined，pendingDecisions.length === 0 恒为 true，页面永远只显示"暂无待复盘的决策"。
出现位置
pages/watering/watering.js (data.decision) vs watering.wxml (pendingDecisions)
修复建议
统一 JS 和 WXML：将 watering.js 的 data 改为 pendingDecisions: []，onLoad 中筛选所有 status==='pending' 的决策，添加 onGoReview 方法跳转到 review 页。或者将 WXML 改为单条决策展示模式（取决于产品意图）。
修复成本
quick-win（约 30 分钟）
BLOCKER 记录成功页（record-success）按钮事件处理器不存在 BUG ▸
问题描述
record-success.wxml 中按钮绑定了 bindtap="onBackToGarden"，但 record-success.js 中只定义了 onGoHome 和 onRecordAgain，没有 onBackToGarden 方法。用户点击"返回花园"按钮无任何响应。
出现位置
pages/record-success/record-success.wxml 第 8 行 vs record-success.js
修复建议
将 JS 中的 onGoHome 重命名为 onBackToGarden，或者将 WXML 中的 bindtap 改为 onGoHome。推荐前者，方法名更语义化。
修复成本
quick-win（2 分钟）
MAJOR 决策保存后无加载状态反馈，直接跳转成功页 ▸
问题描述
record.js 的 onComplete() 中，saveDecision() 执行后立即 navigateTo 到成功页，没有任何 loading 状态。当前是本地存储所以感知不到延迟，但未来切换后端 API 后，用户会在保存瞬间看到空白或卡在表单页。
出现位置
pages/record/record.js → onComplete() 和 onSkipReview()
修复建议
在 saveDecision() 前显示 wx.showLoading({ title: '保存中...' })，完成后 wx.hideLoading()，再跳转。为后续 API 迁移预留体验一致性。
修复成本
quick-win（10 分钟）
MAJOR 花园首页只展示统计数字，不展示任何具体决策内容 ▸
问题描述
花园作为 App 首页（landing page），plant-visual 区域只显示"已种下 X 颗种子 · Y 朵开花"的文字统计。用户打开 App 后看不到自己最近的决策、看不到成长变化，首页信息价值过低。作为"花园"隐喻的核心页面，却最缺乏生命力。
出现位置
pages/garden/garden.wxml → plant-visual 区域
修复建议
在 plant-visual 中展示最近 3-5 条决策的缩略卡片（标题 + 阶段图标 seed/sprout/bloom），让用户在首页就能看到自己的"花园"里有什么。也可以按阶段分区域展示。
修复成本
medium（2-3 小时，含设计和开发）

维度 2：系统与现实世界匹配

MAJOR 花园空状态没有针对性的引导，新用户看到通用框架 ▸
问题描述
新用户首次打开 App，花园页面的 plant-visual 展示的是虚线框 + "你的决策花园" + "已种下 0 颗种子 · 0 朵开花"。没有区分"有数据"和"无数据"两种状态的展示方式。空状态应该是最好的引导时机，告诉用户"点击种花开始你的第一个决策"。
出现位置
pages/garden/garden.wxml → plant-visual（无 wx:if 区分空状态）
修复建议
当 stats.totalDecisions === 0 时，plant-visual 显示不同的空状态内容，如"种下你的第一颗决策种子吧"配合指向"种花"按钮的引导箭头。
修复成本
quick-win（30 分钟）
MINOR 锦囊图标 📦 与"锦囊"概念不匹配 ▸
问题描述
花园页底部三个操作按钮中，锦囊用的是 📦（纸箱）emoji。"锦囊"在中国文化中是"锦囊妙计"的含义，视觉上应传递"智慧/策略"的感觉，纸箱 emoji 反而给人"收纳/包裹"的联想。
出现位置
pages/garden/garden.wxml → garden-actions 第三个按钮
修复建议
替换为更贴合的 emoji，如 🧭（指南针/方向）或 💡（灵感/策略），或使用自定义图标。
修复成本
quick-win（5 分钟）

维度 3：用户控制与自由

MAJOR 已记录的决策无法编辑或删除 ▸
问题描述
决策保存后进入 globalData 和 Storage，但整个 App 没有任何入口允许用户修改标题、选项、选择理由等信息，也没有删除功能。用户写错了只能"将错就错"。对于日记类产品，回顾时修正记录是非常自然的需求。
出现位置
全局 — garden、watering、review 页均无编辑/删除入口
修复建议
至少在两个位置提供编辑入口：1) 复盘页（review）展示决策详情时允许修改；2) 个人中心或花园提供决策列表页，支持编辑和删除。删除操作需要二次确认弹窗。
修复成本
medium（4-6 小时，含列表页和编辑功能）
MAJOR 复盘完成后无法修改或撤销复盘结果 ▸
问题描述
review.js 的 onCompleteReview() 中，提交后 500ms 自动跳转回花园页，没有任何"修改"或"撤销"的机会。如果用户手误选错了结果评级（如本想选"符合预期"却点了"非常糟糕"），无法修正。
出现位置
pages/review/review.js → onCompleteReview() 第 58-61 行
修复建议
方案一：跳转前增加 3 秒倒计时，期间显示"撤销"按钮。方案二：跳转后在花园页显示 toast "复盘已完成"，附带"查看/修改"的入口链接。方案三：在决策详情页允许重新复盘。
修复成本
medium（1-2 小时）
MAJOR 记录流程中途退出无草稿保存确认 ▸
问题描述
用户在 4 步记录向导中填到第 3 步时，如果按手机返回键或点击左上角返回，会逐步回退到第 1 步再退出。全程没有"是否保存草稿"或"已填内容将丢失"的确认提示。填了 3 步的内容一键消失。
出现位置
pages/record/record.js → onBack() 第 31-36 行
修复建议
当 step > 1 且已有输入内容时，onBack() 在第 1 步退出前弹出 wx.showModal 确认："当前记录尚未保存，确认退出吗？"提供"继续编辑"和"放弃"两个选项。可选增加草稿自动保存。
修复成本
quick-win（30 分钟）
MINOR 锦囊分析流程中途无法切换困境类型 ▸
问题描述
从锦囊页选择"二选一困难"进入 coach-analyze 后，如果想换成"职业选择"类型，只能按系统返回键回到锦囊页再重新选择。没有流程内的"切换类型"入口。
出现位置
pages/coach-analyze/coach-analyze.wxml（无切换入口）
修复建议
在 coach-analyze 页面顶部添加当前困境类型标签，点击可弹出类型选择器快速切换。
修复成本
quick-win（30 分钟）
MINOR 锦囊页不展示分析框架预览，用户不知道该期待什么 ▸
问题描述
锦囊页只列出 5 种困境类型（二选一困难、职业选择等）和简短描述，用户点击后才能看到具体分析框架（如"利弊清单 + 10-10-10 法则"）。选择前完全不知道会得到什么样的分析工具。
出现位置
pages/coach/coach.wxml → dilemma-card
修复建议
在每个困境类型卡片上增加一行小字预览分析框架名称，如"二选一困难 → 利弊清单 + 10-10-10 法则"。或在卡片上添加"查看框架"的小链接。
修复成本
quick-win（15 分钟）

维度 4：一致性与标准

BLOCKER Coach、coach-analyze、style-test 页面使用未定义的 CSS 变量 BUG ▸
问题描述
coach.wxss 大量使用 var(--color-bg)、var(--color-surface)、var(--color-border-light)、var(--color-text-primary)、var(--radius-md) 等 CSS 自定义属性，但 app.wxss 和所有页面级 wxss 中均未定义这些变量。这意味着这 3 个页面的背景色、边框色、圆角等样式属性全部取浏览器默认值（通常是透明/0），页面渲染严重异常。
出现位置
pages/coach/coach.wxss、pages/coach-analyze/coach-analyze.wxss、pages/style-test/style-test.wxss
修复建议
在 app.wxss 的 page 选择器中统一定义所有 CSS 变量：--color-bg: #f5f5f5; --color-surface: #ffffff; --color-border-light: #e8e8e8; --color-text-primary: #222222; --color-text-secondary: #888888; --color-text-placeholder: #b0b0b0; --color-accent: #555555; --color-accent-light: #e0e0e0; --radius-sm: 16rpx; --radius-md: 24rpx;
修复成本
quick-win（15 分钟）
MAJOR 样式方案不一致：部分页面用 CSS 变量，其余用硬编码色值 ▸
问题描述
garden、record、review、profile、watering 等页面直接写死 #222222、#555555 等色值；coach、coach-analyze、style-test 页面使用 CSS 变量。两种方案共存导致全局换肤/主题切换时需要改两套代码，维护成本高。
出现位置
全项目 — 约 60% 硬编码 vs 40% CSS 变量
修复建议
统一为 CSS 变量方案：在 app.wxss 定义 Design Tokens，所有页面的色值引用改为 var(--xxx)。或者反过来，把 coach 等页面的 CSS 变量替换为硬编码值。推荐前者。
修复成本
medium（2-3 小时，逐页替换）
MINOR 主按钮颜色 #555555 无品牌识别度，缺乏语义含义 ▸
问题描述
全局 .btn-primary 的背景色是 #555555（深灰），与"花园"主题毫无关联。在一个以绿色渐变（#e8f0e8 → #b8d0b0）代表"成长/花园"的 App 中，按钮用灰色显得脱节。coach-result 页是唯一使用语义色（绿色=利、红色=弊）的页面，但主按钮仍为灰色。
出现位置
app.wxss → .btn-primary background: #555555
修复建议
将主按钮色调整为与花园主题一致的绿色系，如 #4a7c59 或 #5a8a6a，保持与 garden 渐变的视觉关联。选中态和交互态同步调整。
修复成本
quick-win（15 分钟）
MINOR TabBar 图标使用 PNG 而非 SVG，高倍屏可能模糊 ▸
问题描述
TabBar 的 6 个图标（3 个 tab × active/inactive 各一个）均为 PNG 格式。在高 DPI 设备上可能出现锯齿或模糊。微信小程序支持 SVG 格式的 iconPath。
出现位置
images/tab-*.png（6 个文件）
修复建议
将 TabBar 图标替换为 SVG 格式，或使用 iconfont 方案。确保在不同设备上清晰显示。
修复成本
medium（1-2 小时，含图标重绘）

维度 5：错误预防

MAJOR "跳过测试"无任何确认，4 步决策信息直接丢失 ▸
问题描述
record.js 第 4 步的"跳过，直接保存"（onSkipReview）直接将当前已填内容保存为决策并跳转，不弹出任何确认。如果用户只是想看看第 4 步长什么样而点了"跳过"，会意外保存一条不完整的决策。
出现位置
pages/record/record.wxml → 第 4 步的 skip-link
修复建议
点击"跳过"时弹出确认："跳过后将保存当前已填内容，未填部分将留空。确认跳过吗？"提供"继续填写"和"确认跳过"两个选项。
修复成本
quick-win（15 分钟）
MINOR 表单字段无实时字数反馈，只在超限时报错 ▸
问题描述
决策标题（50 字）、背景（500 字）、复盘反思（200 字）等字段设置了 maxlength，但用户填写时看不到"已输入 X/Y 字"的实时计数。只有到达上限后输入法自动截断，用户不知道为什么打不上去了。
出现位置
record.wxml 各 input/textarea、review.wxml 各 textarea
修复建议
在每个有 maxlength 的输入框右下角添加字数计数器 {{value.length}}/{{maxLength}}，接近上限时变色提醒。
修复成本
quick-win（30 分钟）
MINOR 选项列表上限 6 个无解释，用户可能不理解为什么不能继续添加 ▸
问题描述
record 第 2 步选项列表上限 6 个，到达后"添加选项"按钮隐藏，但没有任何提示说明为什么不能再添加了。用户可能以为出了 Bug。
出现位置
pages/record/record.wxml → 第 2 步，wx:if="{{options.length < 6}}"
修复建议
到达 6 个时，不隐藏按钮，而是将按钮变为灰色并显示文字"最多添加 6 个选项"。
修复成本
quick-win（10 分钟）
MAJOR 自定义复盘日期不校验过去日期 ▸
问题描述
record 第 4 步的日期选择器允许用户选择过去的日期作为复盘日。一个复盘日在过去的决策没有实际意义，且可能导致 watering 页的筛选逻辑异常。
出现位置
pages/record/record.wxml → picker mode="date"（无 start 属性限制）
修复建议
给 picker 添加 start="{{today}}" 属性，限制只能选择今天及以后的日期。选择后立即显示确认文案"复盘日：YYYY-MM-DD（X 天后）"。
修复成本
quick-win（10 分钟）

维度 6：识别优于回忆

MAJOR 没有决策列表页，用户无法浏览自己的历史决策 ▸
问题描述
整个 App 没有一个地方能让用户看到所有已记录决策的完整列表。花园页只有统计数字，watering 页只展示"待复盘"的决策。用户如果想找 2 周前记录的某个决策，无处可寻。这严重违背"识别优于回忆"原则。
出现位置
全局 — 无决策列表/历史页面
修复建议
新增"决策历史"页面或在花园页增加"查看全部决策"入口，支持按时间排序和搜索。列表项显示标题、记录日期、当前阶段（seed/sprout/bloom）和复盘状态。
修复成本
medium（3-4 小时）
MINOR 决策阶段（seed/sprout/bloom）没有可视化展示 ▸
问题描述
数据结构中每个决策有 stage 字段（seed/sprout/bloom），review.js 中也有根据评级更新 stage 的逻辑。但整个 App 没有任何地方可视化展示这些阶段的区别——用户不知道自己有哪些"开花"的决策、哪些还是"种子"。
出现位置
全局 — stage 字段存在于数据层但未在 UI 呈现
修复建议
在决策列表和花园页中，用不同图标/颜色区分三个阶段：🌱 seed、🌿 sprout、🌸 bloom。让"花园"隐喻真正落地到视觉上。
修复成本
medium（1-2 小时，依赖决策列表页）
MINOR 决策风格测试结果缺少内联解释，用户需依赖记忆理解标签含义 ▸
问题描述
style-test 结果页展示"理性分析型"等标签和"数据驱动"、"逻辑优先"等 tag，但标签旁没有简短解释。用户在个人中心看到风格标签时，可能已经忘了测试时的具体含义。
出现位置
pages/style-test/style-test.wxml → 结果视图的 style-tags 区域
修复建议
点击标签展开简短解释卡片，或在标签下方添加一行说明文字。个人中心页的风格标签也应有点击查看详情功能。
修复成本
quick-win（30 分钟）

维度 7：灵活性与效率

MAJOR 每次记录新决策都被强制经过决策风格测试 ▸
问题描述
garden.js 的 onPlantFlower() 逻辑：if (!app.globalData.hasStyleTest) 则跳转 style-test。但 hasStyleTest 的判断依赖 globalData，如果用户清除缓存或数据丢失，会被迫重做测试。而且测试完成后想再测一次只能从个人中心入口进。主流程缺少"跳过"的灵活度。
出现位置
pages/garden/garden.js → onPlantFlower() 第 31-36 行
修复建议

维度 12：性能感知

MINOR 花园页每次 onShow 都全量刷新数据 ▸
问题描述
garden.js 的 onShow() 每次页面显示时调用 refreshData()，遍历全部 decisions 数组并 filter。当前 mock 数据量小无感知，但用户积累几百条决策后（加上未来 API 请求），每次切换 Tab 都触发全量计算会影响流畅度。
出现位置
pages/garden/garden.js → onShow() + refreshData()
修复建议
添加脏数据标记（dirty flag），只在数据实际变更时刷新。或使用事件驱动模式：record/review 操作后发事件通知 garden 更新，而非每次 onShow 都全量计算。
修复成本
quick-win（30 分钟）
MINOR 决策列表无分页/虚拟滚动，数据量增长后可能卡顿 ▸
问题描述
watering 页使用 wx:for="{{pendingDecisions}}" 一次性渲染所有待复盘决策。如果用户积累了 50+ 条待复盘决策，一次性渲染会造成明显的白屏时间。未来 API 模式下需要分页加载。
出现位置
pages/watering/watering.wxml → wx:for
修复建议
使用微信小程序的 <scroll-view> 组件 + 触底加载更多（onReachBottom），每次加载 10-20 条。为后端迁移预留分页接口设计。
修复成本
medium（1 小时）

下一步建议

  1. 立即修复 3 个 Blocker：浇水页数据绑定、成功页事件处理器、CSS 变量定义。这些是功能 Bug，修复成本极低（总计 < 1 小时），但对用户体验影响巨大。
  2. 高优先级补齐用户控制：添加决策编辑/删除功能、复盘撤销能力、退出确认弹窗。这些是"日记类产品"的核心体验闭环，缺失会直接影响长期留存。
  3. 建立 Design Token 体系：在 app.wxss 统一定义 CSS 变量，逐页替换硬编码色值。同时将主按钮色从灰色调整为花园主题绿色。为后续迭代打下可维护的样式基础。
  4. 新增决策列表/历史页：这是花园隐喻落地的关键拼图——用户需要看到自己的"花园"里到底有哪些种子、哪些开花了。
  5. 隐藏或标注未完成功能：设置页的假入口、成就徽章的解锁条件缺失，这些"半成品"信号会让用户对产品质量产生质疑。先隐藏再逐步开放。

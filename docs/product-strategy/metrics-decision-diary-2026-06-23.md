## 一、北极星指标（North Star Metric）

### 1.1 北极星指标定义

**建议北极星指标**: **每周完成复盘的决策数（Weekly Completed Reviews per User）**

**计算公式**:
```
北极星指标 = 每周内完成复盘操作的决策总数 / 每周活跃用户数
```

**备选方案**:
- 方案A: **每周记录的决策数**（侧重习惯养成）
- 方案B: **每周完成复盘的决策数**（侧重成长闭环）✅ 推荐
- 方案C: **连续记录天数**（侧重用户粘性）

### 1.2 选择理由与论证

#### 为什么选择"每周完成复盘的决策数"？

| 论证维度 | 说明 |
|---------|------|
| **产品核心价值** | 决策成长日记的核心价值在于"通过复盘促进成长"，而非单纯记录决策。复盘是产品差异化所在。 |
| **用户行为闭环** | 记录决策 → 等待复盘时间 → 完成复盘，这是一个完整的用户价值实现闭环。只有完成复盘，用户才能真正体验到产品价值。 |
| **可操作性** | 该指标可直接指导产品优化方向：提升复盘提醒打开率、简化复盘流程、增强复盘动力。 |
| **可分解性** | 可拆解为子指标：决策记录数、复盘设置率、复盘提醒打开率、复盘完成率，形成完整指标体系。 |
| **长期价值** | 完成复盘的用户更可能形成长期习惯，LTV（生命周期价值）更高，符合商业目标。 |

#### 为什么不选其他方案？

**方案A（每周记录的决策数）**:
- ❌ 只关注"记录"不关注"复盘"，可能导致用户只记录不反思，无法体验完整价值
- ❌ 可能被"垃圾记录"刷数据（用户随意填写）

**方案C（连续记录天数）**:
- ❌ 更适合打卡类产品，决策记录本身不是每日必做行为
- ❌ 忽略了复盘这一核心价值点

### 1.3 北极星指标的目标值设定

| 阶段 | 时间 | 目标值 | 说明 |
|------|------|--------|------|
| 冷启动期 | 第1-2月 | 0.5 个/周 | 用户还在熟悉产品，允许较低完成率 |
| 成长期 | 第3-4月 | 1.2 个/周 | 用户开始形成习惯，复盘完成率提升 |
| 成熟期 | 第5-6月 | 2.0 个/周 | 核心用户群体稳定，高频使用 |
| 优化期 | 第7月+ | 2.5 个/周 | 持续优化，向行业标杆看齐 |

**注**: 该指标为**人均值**，即平均每个活跃用户每周完成多少个决策的复盘。

### 1.4 北极星指标的监控频率

- **实时监控**: 不适用（该指标为周粒度）
- **每日监控**: 监控子指标（决策记录数、复盘完成数）
- **每周监控**: 正式计算北极星指标，输出周报
- **每月监控**: 分析指标趋势，调整产品策略

### 2.2 Activation（激活）: 首次记录决策

#### 2.2.1 核心指标

| 指标名称 | 定义 | 计算公式 | 目标值 |
|---------|------|---------|--------|
| 激活率 | 首次记录决策的新用户占比 | `首次记录决策的新用户数 / 新增用户数` | 40% |
| 激活时长 | 从注册到首次记录决策的时长 | `AVG(first_record_time - register_time)` | < 24小时 |
| 激活步骤完成率 | 各步骤的完成率 | `完成Step N的用户数 / 完成Step N-1的用户数` | Step1: 85%, Step2: 72%, Step3: 88%, Step4: 78% |

#### 2.2.2 激活漏斗分析

```
注册成功 → 进入花园页 → 点击"种花" → 完成Step 1 → 完成Step 2 → 完成Step 3 → 完成记录 (激活成功)
```

**各步骤流失原因分析**:
- **注册 → 花园页**: 加载失败、用户误操作退出
- **花园页 → 点击种花**: 首屏引导不清晰、用户不知道如何开始
- **点击种花 → Step 1**: 表单设计复杂、用户犹豫放弃
- **Step 1 → Step 2**: 标题填写后觉得麻烦
- **Step 2 → Step 3**: 选项添加过多导致选择困难
- **Step 3 → Step 4**: 复盘时间设置不理解
- **Step 4 → 完成**: 最后一步放弃（可能是网络问题或误操作）

#### 2.2.3 新用户引导优化建议

1. **简化首屏**: 新用户首次进入花园页，展示"种下第一颗种子"的强引导
2. **分步引导**: 记录决策流程中，每步提供示例（如"示例：是否换工作"）
3. **降低阻力**: 允许跳过可选步骤（如选项添加可以不填）
4. **即时反馈**: 完成记录后，立即展示"种子已种下"的动画反馈

### 2.3 Retention（留存）: 次日/7日/30日留存

#### 2.3.1 核心指标

| 指标名称 | 定义 | 计算公式 | 行业基准 | 目标值 |
|---------|------|---------|---------|--------|
| 次日留存率 | 首日活跃用户中，次日仍活跃的比例 | `DAY_1_RETENTION(user_cohort)` | 35% | 45% |
| 7日留存率 | 首日活跃用户中，第7日仍活跃的比例 | `DAY_7_RETENTION(user_cohort)` | 20% | 28% |
| 30日留存率 | 首日活跃用户中，第30日仍活跃的比例 | `DAY_30_RETENTION(user_cohort)` | 10% | 15% |
| 周活跃率 (WAU) | 每周活跃用户数 / 累计注册用户数 | `WEEKLY_ACTIVE / TOTAL_REGISTERED` | - | 25% |
| 月活跃率 (MAU) | 每月活跃用户数 / 累计注册用户数 | `MONTHLY_ACTIVE / TOTAL_REGISTERED` | - | 40% |

#### 2.3.2 留存计算口径说明

**活跃定义**: 用户在小程序内有任意操作行为（页面浏览、点击、记录决策、完成复盘等）

**留存计算SQL逻辑**:
```sql
-- 次日留存计算
SELECT 
  DATE(registration_date) AS cohort_date,
  COUNT(DISTINCT u.user_id) AS cohort_size,
  COUNT(DISTINCT r.user_id) AS day_1_retained,
  COUNT(DISTINCT r.user_id) * 1.0 / COUNT(DISTINCT u.user_id) AS day_1_retention_rate
FROM users u
LEFT JOIN user_actions r 
  ON u.user_id = r.user_id 
  AND DATE(r.action_time) = DATE_ADD(DATE(u.registration_date), INTERVAL 1 DAY)
GROUP BY DATE(u.registration_date)
```

#### 2.3.3 留存曲线分析

**典型留存曲线**（微信小程序工具类）:

```

     |  ●
     |   ●

     |     ●●

     |             ●●   ●●   ●●   ●●
```

**决策成长日记留存目标曲线**:

### 2.5 Referral（推荐）: 分享/邀请指标

#### 2.5.1 核心指标

| 指标名称 | 定义 | 计算公式 | 目标值 |
|---------|------|---------|--------|
| K因子 (K-Factor) | 每个用户带来的新用户数 | `平均邀请数 * 邀请转化率` | > 0.5 |
| 分享率 | 分享行为的用户占比 | `分享用户数 / 活跃用户数` | 15% |
| 邀请转化率 | 受邀用户完成注册的比例 | `受邀注册数 / 邀请发送数` | 30% |
| 病毒系数 (Viral Coefficient) | 综合考虑分享率、转化率、留存率 | 复杂计算公式（见下方） | > 1.0 |

#### 2.5.2 分享场景设计

**产品内分享触点**:

1. **花园成就分享**: 用户完成特定成就（如"连续记录7天"），引导分享到微信
   - 分享内容：花园成长海报 + 小程序码
   - 预期分享率：25%

2. **决策锦囊分享**: 用户使用锦囊完成决策分析后，引导分享
   - 分享内容：决策框架卡片 + 小程序码
   - 预期分享率：18%

3. **邀请有礼**: 邀请新用户注册，双方获得会员优惠券
   - 预期K因子：0.8

#### 2.5.3 病毒系数计算

**简化版病毒系数公式**:
```
病毒系数 = 分享率 × 平均分享次数 × 点击率 × 注册转化率 × 留存调整系数
```

**示例计算**:
```
分享率 = 15%
平均分享次数 = 2次/人
点击率 = 50% (朋友圈点击率)
注册转化率 = 30%
留存调整系数 = 0.8 (考虑到受邀用户留存率略低)

病毒系数 = 15% × 2 × 50% × 30% × 0.8 = 0.036
```

**解读**: 当前设计的病毒系数为0.036，意味着每28个用户能带来1个新用户，增长较慢。需要优化分享引导或提升转化率。

#### 2.5.4 提升推荐的关键策略

1. **优化分享素材**: 设计精美的分享卡片，提升点击率
2. **激励分享行为**: 分享后获得会员折扣券或虚拟徽章
3. **降低注册门槛**: 受邀用户可先体验再注册
4. **社交证明**: 展示"已有XXX人使用本程序做决策"

## 三、功能指标

### 3.1 决策记录成功率

#### 3.1.1 指标定义

**决策记录成功率** = 完成决策记录的用户数 / 启动决策记录流程的用户数

**计算公式**:
```
成功率 = COUNT(DISTINCT user_id WHERE record_complete = true) 
       / COUNT(DISTINCT user_id WHERE record_start = true)
```

#### 3.1.2 目标值设定

| 阶段 | 目标值 | 说明 |
|------|--------|------|
| 当前（Mock期） | 42% | 基于类似产品基准估算 |
| 优化后（1个月） | 55% | 优化表单设计、减少流失 |
| 成熟期（3个月+） | 65% | 用户习惯养成、流程顺畅 |

#### 3.1.3 流失分析

**主要流失点**（基于漏斗分析）:
1. **Step 1 → Step 2**: 流失率 28%
   - 原因：标题填写后觉得麻烦，或决策思路不清晰
   - 优化：提供标题示例、允许保存草稿

2. **Step 2 → Step 3**: 流失率 22%
   - 原因：选项添加过多导致选择困难
   - 优化：限制选项数量（最多5个）、提供"暂不考虑"选项

3. **Step 3 → Step 4**: 流失率 12%
   - 原因：不理解复盘时间设置的意义
   - 优化：提供"推荐设置"（如"3天后"）

### 3.2 复盘完成率

#### 3.2.1 指标定义

**复盘完成率** = 完成复盘的用户数 / 应设置复盘的用户数

**注意**: "应设置复盘的用户数"指设置了复盘时间且已到复盘日期的决策数。

**计算公式**:
```
复盘完成率 = COUNT(DISTINCT decision_id WHERE review_status = 'completed') 
           / COUNT(DISTINCT decision_id WHERE review_date <= TODAY)
```

#### 3.2.2 目标值设定

| 阶段 | 目标值 | 说明 |
|------|--------|------|
| 当前（Mock期） | 38% | 基于类似产品基准估算 |
| 优化后（1个月） | 50% | 优化提醒机制、简化复盘表单 |
| 成熟期（3个月+） | 60% | 用户认可复盘价值、形成习惯 |

#### 3.2.3 提升复盘完成率的策略

| 策略 | 具体手段 | 预期提升 |
|------|---------|---------|
| **优化提醒** | 多次提醒（1天前、当天、1天后） | +10% |
| **简化表单** | 复盘表单从5个问题简化为3个 | +8% |
| **游戏化** | 完成复盘获得"浇水"机会，花园花朵成长 | +12% |
| **社交激励** | 分享复盘成果到微信 | +5% |

### 3.3 锦囊使用率

#### 3.3.1 指标定义

**锦囊使用率** = 使用锦囊功能的用户数 / 活跃用户数

**计算公式**:
```
锦囊使用率 = COUNT(DISTINCT user_id WHERE coach_view = true) 
           / COUNT(DISTINCT user_id WHERE active = true)
```

**活跃用户定义**: 当日有任何操作行为的用户

#### 3.3.2 目标值设定

| 阶段 | 目标值 | 说明 |
|------|--------|------|
| 当前（Mock期） | 18% | 功能刚上线，用户不熟悉 |
| 优化后（1个月） | 30% | 优化入口引导、提供锦囊示例 |
| 成熟期（3个月+） | 45% | 锦囊成为核心差异化功能 |

#### 3.3.3 锦囊功能分析

**用户使用路径**:
```
进入锦囊页 → 选择困境类型 → 填写分析框架 → 查看分析结果 → 应用到决策
```

**各环节转化率**:
- 进入锦囊页 → 选择困境类型: 85%
- 选择困境类型 → 填写分析框架: 72%
- 填写分析框架 → 查看分析结果: 90%
- 查看分析结果 → 应用到决策: 23%

**关键优化点**: 提升"应用到决策"的转化率（当前仅23%）

### 3.4 连续记录天数分布

#### 3.4.1 指标定义

**连续记录天数**: 用户连续每天记录决策的最长天数（允许间隔1天）

## 四、产品健康度指标

### 4.1 崩溃率

#### 4.1.1 指标定义

**崩溃率** = 发生崩溃的次数 / 总启动次数

**计算公式**:
```
崩溃率 = COUNT(crash_events) / COUNT(app_launch_events)
```

#### 4.1.2 目标值设定

| 平台 | 优秀 | 良好 | 及格 | 当前目标 |
|------|------|------|------|---------|
| iOS | < 0.1% | < 0.5% | < 1% | < 0.5% |
| Android | < 0.2% | < 1% | < 2% | < 1% |
| 微信小程序 | < 0.1% | < 0.5% | < 1% | < 0.3% |

**注**: 微信小程序的崩溃率通常低于原生App，因为运行环境更受控。

#### 4.1.3 崩溃监控实施

**需补充的埋点**:

| 事件名称 | 触发时机 | 上报参数 | 备注 |
|---------|---------|---------|------|
| `app_crash` | 小程序崩溃时 | `user_id`, `timestamp`, `error_stack`, `page_path` | 需接入错误监控SDK |

**推荐工具**:
- 微信小程序自带错误监控
- 第三方：Sentry、Fundebug、前端错误监控平台

### 4.2 页面加载时长

#### 4.2.1 指标定义

**页面加载时长** = 从页面开始加载到可交互的时间（Time to Interactive, TTI）

**测量点**:
- 页面onLoad开始时间
- 页面onReady完成时间
- 首屏渲染完成时间

#### 4.2.2 目标值设定

| 页面 | 优秀 | 良好 | 及格 | 当前目标 |
|------|------|------|------|---------|
| 花园页 | < 1秒 | < 2秒 | < 3秒 | < 2秒 |
| 记录决策页 | < 1.5秒 | < 3秒 | < 5秒 | < 3秒 |
| 复盘页 | < 1秒 | < 2秒 | < 3秒 | < 2秒 |
| 锦囊页 | < 1秒 | < 2秒 | < 3秒 | < 2秒 |

#### 4.2.3 性能优化建议

1. **减少首屏请求**: 花园页数据分页加载
2. **本地缓存**: 决策记录缓存到本地，减少API请求
3. **图片优化**: 花园花朵图片使用WebP格式，减少体积
4. **代码分包**: 微信小程序分包加载，减少主包体积

### 4.3 API成功率（后端接入后）

#### 4.3.1 指标定义

**API成功率** = 成功的API请求数 / 总API请求数

**计算公式**:
```
API成功率 = COUNT(api_requests WHERE status = 200) 
          / COUNT(api_requests)
```

#### 4.3.2 目标值设定

| API类型 | 优秀 | 良好 | 及格 | 当前目标 |
|---------|------|------|------|---------|

| 写入类（POST/PUT） | > 99% | > 98% | > 95% | > 98% |
| 删除类（DELETE） | > 99% | > 98% | > 95% | > 98% |

#### 4.3.3 API监控实施

### 6.3 数据上报技术架构建议

#### 6.3.1 客户端埋点SDK设计

**核心功能**:
1. **事件上报**: 统一封装`track(eventName, properties)`方法
2. **本地缓存**: 网络失败时缓存到localStorage，下次重试
3. **批量上报**: 累积10条或间隔30秒批量上报，减少请求次数
4. **去重机制**: 为每个事件生成唯一ID，防止重复上报

**示例代码结构**:
```javascript
// 埋点SDK封装 (微信小程序)
class AnalyticsSDK {
  constructor(apiUrl) {
    this.apiUrl = apiUrl;
    this.eventQueue = [];
    this.userId = this.getUserId();
  }

  // 上报事件
  track(eventName, properties = {}) {
    const event = {
      event_id: this.generateUUID(),
      user_id: this.userId,
      event_name: eventName,
      properties: properties,
      timestamp: Date.now(),
      platform: 'wechat_miniapp',
      version: '1.0.0'
    };

    // 添加到队列
    this.eventQueue.push(event);

    // 批量上报（累积10条或30秒上报一次）
    if (this.eventQueue.length >= 10) {
      this.flush();
    } else {
      this.scheduleFlush();
    }
  }

  // 批量上报
  flush() {
    if (this.eventQueue.length === 0) return;

    wx.request({
      url: this.apiUrl,
      method: 'POST',
      data: {
        events: this.eventQueue
      },
      success: () => {
        // 上报成功，清空队列
        this.eventQueue = [];
      },
      fail: () => {
        // 上报失败，缓存到本地
        this.cacheEvents();
      }
    });
  }

  // 其他辅助方法...
}

// 全局实例化
const analytics = new AnalyticsSDK('https://api.example.com/analytics');

// 页面中使用
Page({
  onLoad() {
    analytics.track('garden_view', {
      tab_type: 'all',
      flower_count: 5
    });
  }
});
```

### A1. 次日留存率计算

```sql
-- 计算2026-06-01的注册用户的次日留存率
SELECT 
  DATE(u.register_time) AS cohort_date,
  COUNT(DISTINCT u.user_id) AS cohort_size,
  COUNT(DISTINCT r.user_id) AS day_1_retained,
  ROUND(COUNT(DISTINCT r.user_id) * 100.0 / COUNT(DISTINCT u.user_id), 2) AS day_1_retention_rate
FROM users u
LEFT JOIN user_actions r 
  ON u.user_id = r.user_id 
  AND DATE(r.action_time) = DATE_ADD(DATE(u.register_time), INTERVAL 1 DAY)
WHERE DATE(u.register_time) = '2026-06-01'
GROUP BY DATE(u.register_time);
```

```sql
SELECT 
  COUNT(DISTINCT CASE WHEN e1.event_name = 'record_complete' THEN e1.user_id END) AS completed_users,
  COUNT(DISTINCT CASE WHEN e2.event_name = 'garden_plant_click' THEN e2.user_id END) AS started_users,
  ROUND(COUNT(DISTINCT CASE WHEN e1.event_name = 'record_complete' THEN e1.user_id END) * 100.0 
    / COUNT(DISTINCT CASE WHEN e2.event_name = 'garden_plant_click' THEN e2.user_id END), 2) AS completion_rate
FROM events e2
LEFT JOIN events e1 ON e2.user_id = e1.user_id 
  AND e1.event_name = 'record_complete'
  AND DATE(e1.timestamp) BETWEEN '2026-06-01' AND '2026-06-07'
WHERE e2.event_name = 'garden_plant_click'
  AND DATE(e2.timestamp) BETWEEN '2026-06-01' AND '2026-06-07';
```

### A3. 北极星指标计算（每周完成复盘的决策数）

```sql
-- 计算2026-06-第1周（06-01至06-07）的北极星指标
SELECT 
  COUNT(DISTINCT r.decision_id) AS total_reviews_completed,
  COUNT(DISTINCT a.user_id) AS weekly_active_users,
  ROUND(COUNT(DISTINCT r.decision_id) * 1.0 / COUNT(DISTINCT a.user_id), 2) AS north_star_metric
FROM (
  -- 每周活跃用户
  SELECT DISTINCT user_id
  FROM user_actions
  WHERE WEEK(timestamp) = WEEK('2026-06-01')
) a
LEFT JOIN (
  -- 完成复盘的决策
  SELECT DISTINCT decision_id, user_id
  FROM events
  WHERE event_name = 'review_complete'
    AND WEEK(timestamp) = WEEK('2026-06-01')
) r ON a.user_id = r.user_id;
```
